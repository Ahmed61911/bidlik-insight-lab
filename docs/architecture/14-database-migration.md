# 14 — Database Migration

## Current shape

- **Engine**: PostgreSQL 15.6 with `pg_cron`, `pgcrypto`, `uuid-ossp`,
  `pg_stat_statements`, `pgaudit`.
- **ORM**: none in TypeScript — everything goes through PostgREST
  (auto-generated REST) and hand-written PL/pgSQL RPCs.
- **Migrations**: raw SQL files under `supabase/migrations/*.sql`, applied
  by `docker/migrate/run.sh` and tracked in `public._app_migrations`.
- **Connection management**: PostgREST uses a Postgres connection pool
  (`authenticator` role); GoTrue and Storage use their own pools.
- **Repository pattern**: none — the "repository" is Postgres itself.
- **Transactions**: implicit per RPC; explicit `SELECT ... FOR UPDATE`
  inside `place_bid` and `submit_offer`.

### DbContext sketch

```csharp
public class BidlicDbContext : DbContext {
    public DbSet<Auction> Auctions => Set<Auction>();
    public DbSet<Bid> Bids => Set<Bid>();
    public DbSet<Offer> Offers => Set<Offer>();

    public DbSet<Profile> Profiles => Set<Profile>();

    public DbSet<Payment> Payments => Set<Payment>();
    public DbSet<Notification> Notifications => Set<Notification>();
    public DbSet<ExpertAssignment> ExpertAssignments => Set<ExpertAssignment>();

    protected override void OnModelCreating(ModelBuilder mb) {
        mb.HasPostgresEnum<AppRole>();       // Npgsql native enum support
        mb.HasPostgresEnum<AuctionStatus>(); // repeat for every enum
        // ...
        mb.ApplyConfigurationsFromAssembly(typeof(BidlicDbContext).Assembly);
    }

```

Every enum stays as a native Postgres enum (matches the current schema).
Ownership filters ("current user only") live in a global query filter or
in each service:

### RPC replacement — Dapper sidebar

`place_bid` cannot be safely translated to EF Core because EF's
`ExecuteUpdate` cannot express `SELECT ... FOR UPDATE`. Dapper version:

```csharp
public async Task<Bid> PlaceBidAsync(string auctionId, int amount, bool isAuto = false) {
    await using var conn = new NpgsqlConnection(_cs);
    await conn.OpenAsync();
    await using var tx = await conn.BeginTransactionAsync();

    var a = await conn.QuerySingleOrDefaultAsync<Auction>(
        "SELECT * FROM auctions WHERE id=@id FOR UPDATE", new { id = auctionId }, tx)
        ?? throw new NotFoundException();

    if (a.Status != AuctionStatus.Live) throw new ConflictException("Auction not live");
    if (amount <= a.CurrentPrice) throw new ValidationException($"> {a.CurrentPrice}");

    var newEnd = (a.EndsAt - DateTime.UtcNow) < TimeSpan.FromMinutes(2)
        ? DateTime.UtcNow.AddMinutes(2)
        : a.EndsAt;

    var bid = await conn.QuerySingleAsync<Bid>(@"
        INSERT INTO bids(auction_id, car_id, bidder_id, bidder_name, amount, is_auto)
        VALUES(@AuctionId,@CarId,@BidderId,@BidderName,@Amount,@IsAuto)
        RETURNING *;", new { ... }, tx);

    await conn.ExecuteAsync(@"
        UPDATE auctions
           SET current_price=@Amount, bid_count=bid_count+1,
               top_bidder_id=@BidderId, ends_at=@NewEnd
         WHERE id=@Id;",
        new { Amount = amount, BidderId = _ctx.UserId, NewEnd = newEnd, Id = auctionId }, tx);

    await tx.CommitAsync();
    await _hub.Clients.Group($"auction:{auctionId}").SendAsync("BidPlaced", bid);
    return bid;
}
```

Same pattern for `submit_offer`, `tick_auctions`, `buyer_submit_payment`.

## Migration path from current schema

Two options:

### A. Preserve the schema (recommended)

1. Point EF Core's `MigrationHistoryTable` at a new table
   `__EFMigrationsHistory` alongside `_app_migrations`.
2. Import the existing schema into `Data/Migrations/` as an initial
   migration generated by `dotnet ef dbcontext scaffold`.
3. Delete the SECURITY DEFINER RPCs (replaced by services).
4. Optionally drop RLS if you trust the service layer; otherwise keep
   RLS and connect Dapper with a token-bearing role.

### B. Fresh schema, data copy

1. `dotnet ef migrations add Initial` from clean entities.
2. Data pump: `pg_dump --data-only public.*` and `psql` into the new DB.
3. Rewrite enums exactly (Npgsql `HasPostgresEnum` guarantees the wire
   format matches).

## Alternative databases

### SQL Server (`Microsoft.EntityFrameworkCore.SqlServer`)

| Concern | Change |
|---------|--------|
| Enums | Postgres native enums → SQL Server: `nvarchar(50)` + EF value converter, or a `Ref` table with a FK. |
| `jsonb` (`cars.images`) | `NVARCHAR(MAX)` + `HasConversion<string>` (JSON functions available on SQL Server 2022+). |
| `SELECT ... FOR UPDATE` | `SELECT ... WITH (UPDLOCK, ROWLOCK)`. Dapper query changes accordingly. |
| `pg_cron` | Removed — use Hangfire recurring jobs. |
| Text `id` columns | Keep as `NVARCHAR(64)` (client-supplied slugs). |
| UUID | `UNIQUEIDENTIFIER`. |
| Case-insensitive collation | Set at DB level. |

Files affected: `Data/BidlicDbContext.cs` (`UseSqlServer`),
`Repositories/Dapper/*` (query syntax), `Data/Migrations/*` (regenerate),
`docker-compose.yml` (swap `postgres` for `mcr.microsoft.com/mssql/server`).

### MySQL (`Pomelo.EntityFrameworkCore.MySql`)

| Concern | Change |
|---------|--------|
| Enums | MySQL enums exist but are painful; prefer `VARCHAR` + converter. |
| `jsonb` | `JSON` column (MySQL 8+). |
| `FOR UPDATE` | Native, works identically. |
| `pg_cron` | Removed — Hangfire. |
| UUID | `CHAR(36)` or `BINARY(16)`. |
| Full-text search (if added later) | Different syntax. |

Files affected: same as SQL Server, plus MySql-specific migration output.
